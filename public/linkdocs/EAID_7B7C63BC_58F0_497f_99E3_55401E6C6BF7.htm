Introduction

CompassAE Technology Foundations


CompassAE is a fusion of solutions from several bodies of work considered foundational:
  
  • Enterprise Archetypes and Data Model Patterns
  • Rapid and Agile Development Methods Support
  • Modern Deployment Technologies: The Cloud, Web and Device Enablement

The goal of this particular synthesis is to improve the trade-offs between delivery velocity on the one hand, and on the other, flexibility at the application and enterprise architecture levels - between short and long-term concerns facing a design team.

CompassAE Application Architecture


Enterprise Archetypes and Data Model Patterns


CompassAE started as a series of tutorials describing implementation options for Len Silverston’s Data Model Resource Books, and later incorporated ideas from Jim Arlow and Ila Neustadt’s Entperise Archetypes. Example implementation alternatives and code from these tutorials grew and became useful as a starting point for development projects in a variety of circumstances.

The assumption was that starting with tested implementations of these mature models would offset the tendency to simplify the data and object-relational layers of project work to meet deadlines.

Simple user interfaces were then built to allow teams to see how their data would look organized according to the patterns. This helped teams visualize, for example, how customer and supplier data could be stored using Business Party, Role and Relationship structures instead of creating a Customer or Supplier table with fixed roles and relationships.

Rapid and Agile Development Methods Support


Using patterns in general represents an orientation towards upfront investment in longer-term stability and adaptability as opposed to quick results.

The time horizons imposed on IT by the pace, risk tolerance and attention span of today’s businesses make this a very difficult sell for architects who are advocating for the long term.

CompassAE’s application infrastructure started by simply using various scaffolding techniques and the velocity achievable through the Rails framework. Prototypes evolved in parallel from the data layer using scaffolds, and the UI layer using standard user-centered agile techniques. Rails plugins and rubygems were later added to provide default implementations of common application and technical services, such as authentication/authorization, image processing and search indexing.

Over the course of time, an application architecture began to take shape that organized and extended these facilities, and added a container and component model. The hope was to allow an implementation to follow a path that seems to offer reasonable trade-offs between short and long-term benefits. A second goal that has been pursued is that the framework should “get out of the way” if an implementer chooses a different approach in some areas than the framework designers chose.

Modern Deployment Technologies: The Cloud, Web and Device Enablement


If patterns-orientation and model-driven architecture (MDA) are the buzzwords of the moment for architects and designers, and Agile Development is today’s goal for project sponsors and their teams - then the Cloud, Web and Device Enablement is the rage for entrepreneurs and executives at all levels.

This would be interesting enough, except that our belief is that these terms and several other key developments aren’t empty hype. They refer to a fundamental shift in deployment topology that is on the scale of disruption caused by the Client/Server revolution and even the Web itself. Frankly, we like these kinds of disruptions. They keep the world interesting for us techies.

So we started semi-early on tracking the trends associated with these shifts and embraced the idea of HTML5 using Extjs and Sencha for UI technologies and REST as a design principle for web services. We later began to introduce multi-tenant facilities into the core framework. All of this is still very active in the framework development work.

However, this is a point where it makes some sense to go back to the framework’s initial foundation for a coda.

For the “programmable web” to evolve towards the “interconnected, component-based and device-enabled cloud”, there has to be a Rosetta Stone of sorts. The domain language of people, agreements they sign, orders they place and products they buy will need to include a set of base canonical models that trend towards stability. We think that the Enterprise Archetypes and Data Model Patterns work represents the best starting point for such a Rosetta Stone in existence today, and it has the benefit of being published, easily accessible, current and actively evolving, as well as being built for extension and customization where unique problems are still yet to be solved.

These ideas are DNA that shapes the realization of CompassAE.
